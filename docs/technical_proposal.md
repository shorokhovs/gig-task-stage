# Technical Proposal: Global Consolidation Initiative

## Part 1: Architectural Strategy and Migration

### Target Architecture

The target direction is to use PostgreSQL on AWS RDS as the main platform for
new workloads. Legacy MySQL, SQL Server, and older PostgreSQL systems should be
managed as migration sources. The migration process should be repeatable:
discovery, compatibility check, schema conversion, data migration, validation,
cutover, and rollback planning.

For a high-traffic brand, I would not start with a direct production cutover. First, I
would create a production-like PostgreSQL RDS environment. Then I would test the
main workload, query plans, data quality, replication lag, monitoring, backup,
and operational runbooks. Only after that would I move customer traffic.

### Schema Evolution

MySQL-specific features need clear mapping to PostgreSQL. A simple automatic
conversion is not enough.

- `SET` types should usually become a normalized child table. A PostgreSQL array
  with a check constraint can also work, but normalization is better when the
  values are queried often.
- `ENUM` can become PostgreSQL `ENUM` only when values are stable. If values
  change often, a lookup table is easier to manage.
- `AUTO_INCREMENT` maps to `generated by default as identity`.
- `TINYINT(1)` must be checked. It may mean a boolean value or a small number.
- MySQL unsigned integers need either PostgreSQL larger types or check
  constraints.
- Invalid MySQL dates like `0000-00-00` must be cleaned before migration.
- MySQL case-insensitive collations need a PostgreSQL decision, for example
  a specific collation or `citext`.
- AWS DMS table mapping can help with simple data type changes during migration.
  DMS transformation rules support the `change-data-type` action for columns.
  I would still test these rules carefully for precision, scale, length,
  truncation, and application compatibility. Official documentation:
  [Transformation rules and actions](https://docs.aws.amazon.com/dms/latest/userguide/CHAP_Tasks.CustomizingTasks.TableMapping.SelectionTransformation.Transformations.html).
- Locking behavior must be tested. PostgreSQL MVCC and row-level locks are not
  the same as MySQL/InnoDB gap locks. Important write paths need concurrency
  tests and deadlock monitoring.

The result of this phase should be a compatibility matrix, not only converted
DDL. Every important mapping should have an owner, test results, and sign-off.

For DDL conversion, I would use the AWS Schema Conversion Tool (AWS SCT) as the
first step. I would not hand-write the whole target schema from zero. The
assessment must include stored procedures, functions, triggers, events, and views. These objects often contain more engine-specific logic
than tables.

If these objects exist, I would convert them with AWS SCT, review the converted
PostgreSQL code, and rewrite unsupported parts manually. The corrected schema
and server-side objects should be deployed to PostgreSQL first. After the target
is ready, I would run AWS DMS for full-load and CDC data movement. In this
model, SCT prepares the schema and DMS moves the data.
[What is the AWS Schema Conversion Tool?](https://docs.aws.amazon.com/SchemaConversionTool/latest/userguide/CHAP_Welcome.html).

### Zero-Downtime Migration

The cutover should have very short downtime:

1. Freeze schema changes or approve them through a migration board.
2. Convert and deploy the PostgreSQL schema.
3. Prepare the target for full load. If indexes, foreign keys, check
   constraints, or triggers were created during schema conversion, I would
   disable or postpone the non-critical ones before loading data with DMS. This
   reduces write overhead during the initial load. I would keep only the objects
   needed for routing and basic data safety, for example partition constraints
   or required primary keys.
4. Load historical data with DMS full load.
5. Rebuild indexes, enable triggers, and validate constraints after the full
   load. This step must finish before cutover.
6. Start continuous change replication from MySQL to PostgreSQL.
7. Validate data by table, time window, and business totals.
8. Run shadow reads against PostgreSQL where possible.
9. Stop writes for a short time, wait for replication lag to reach zero,
   validate final counts, switch writes to PostgreSQL, and keep rollback scripts
   ready.

AWS DMS is a good default for MySQL to PostgreSQL migration. It supports full
load plus CDC, has lag metrics, and reduces operational work. The weak points
are complex type conversion, edge cases, LOB/JSON behavior, and schema drift.
These areas must be tested before production cutover.

Native logical replication is useful when both sides are PostgreSQL or when the
team already owns a strong replication platform. For MySQL to PostgreSQL, this
usually means using Debezium, Kafka Connect, or a custom binlog pipeline. This
gives more control, but it also creates more platform ownership. I would choose
DMS for the first migration wave.

### Hybrid Connectivity

Connectivity between on-prem MySQL and AWS RDS should use private networking:

- AWS Direct Connect as the main connection for stable latency and throughput.
- Site-to-Site VPN as backup or as a first step.
- Private subnets for RDS.
- Security groups limited to migration workers and application subnets.
- Network ACLs and firewalls based on least privilege.
- TLS for database connections.
- AWS Secrets Manager or another vault for credentials.
- Route 53 private hosted zones for stable internal names.

For AWS DMS, the migration worker is the DMS replication instance. I would place
this replication instance in AWS, in the same VPC and close to the target RDS
PostgreSQL instance. It should connect to on-prem MySQL over Direct Connect or
VPN using private routing. A custom EC2 worker would be used only for a
non-DMS approach, for example Debezium or a custom binlog pipeline. The network
design should be tested with production-like CDC traffic, not only with ping
tests.

## Part 2: Performance and Scalability

### PostgreSQL Purging Under an IOPS Ceiling

If purge jobs hit the IOPS limit, the purge process is too heavy for the storage
configuration. The best long-term solution is time-based partitioning. Old data
can then be archived and dropped by partition. Dropping or detaching a partition
is much cheaper than deleting millions of rows.

For existing non-partitioned tables, I would use small ordered batches, sleep
between batches, and progress tracking. Each job should have a maximum runtime.
The purge predicate, usually `created_at`, must be indexed. Large deletes should
avoid long transactions because they increase WAL, replication lag, lock time,
and autovacuum pressure.

Recommended changes:

- Increase storage class or provisioned IOPS if the workload really needs it.
- Run purge jobs outside peak traffic.
- Use partitioning for new large time-based tables.
- Tune autovacuum for hot tables with lower scale factors and suitable cost
  limits.
- Monitor dead tuples, WAL generation, replica lag, disk queue depth, and
  `pg_stat_user_tables`.
- Use read replicas for reporting if analytical queries add pressure.
- Use `statement_timeout`, `lock_timeout`, and application retries so purge jobs
  do not block front-end traffic for too long.

### RDS Proxy vs PgBouncer

RDS Proxy is a good fit for AWS-native workloads. It works well with Secrets
Manager, IAM authentication, managed failover, and spiky application traffic. It
also reduces the amount of proxy infrastructure that the team must operate.

PgBouncer is a good fit when the team needs more control over pooling behavior.
It is especially useful for transaction pooling and high connection counts. It
is also portable outside AWS. The trade-off is that the team must operate and
monitor PgBouncer. The application must also be compatible with the selected
pooling mode, especially around session state and prepared statements.

For 10,000+ concurrent connections, I would first reduce unnecessary database
sessions in the application. Then I would use PgBouncer for efficient
transaction pooling where the application supports it. I would use RDS Proxy
when managed AWS integration and failover handling are more important.

## Part 3: Automation, Security and Governance

### Infrastructure as Code

I would use a layered IaC model. For AWS-native deployments, CloudFormation or
AWS CDK should be the default option. They integrate directly with IAM, Change
Sets, StackSets, drift detection, and AWS service updates. Terraform is also a
good option when the estate is hybrid or multi-provider, for example AWS plus
on-prem VMs, external DNS, external monitoring, or non-AWS secrets tools.
Ansible should handle host configuration, database bootstrap, and operational
tasks that must run inside the server or database engine.

The infrastructure layer, whether it is CloudFormation/CDK or Terraform, should
offer a common database service interface for PostgreSQL, MySQL, and SQL Server:

- Engine and version.
- Environment, region, network placement, and tags.
- Instance class, storage type, storage size, IOPS, and autoscaling limits.
- Backup retention, maintenance window, deletion protection, and restore policy.
- Encryption with KMS keys.
- Parameter groups or engine configuration profiles.
- Security groups, subnet groups, private DNS, and allowed client networks.
- Monitoring, logs, alarms, and audit integration.
- Secrets Manager or vault references for credentials.

Behind this common interface, I would keep separate engine-specific modules or
constructs. PostgreSQL, MySQL, and SQL Server should not be forced into one
generic module. They have different parameters, HA options, logging options,
extensions, and maintenance behavior. A small catalog layer can define approved
patterns such as `pg-standard`, `mysql-legacy-compatible`, and `sqlserver-ha`.
Each pattern then maps to the correct engine-specific module.

DDL standardization is a separate topic from infrastructure standardization. I
would not try to keep one abstract DDL that generates perfect MySQL,
PostgreSQL, and SQL Server schemas. These engines have different data types,
identity behavior, partitioning syntax, JSON support, indexes, locking behavior,
and procedural languages. A lowest-common-denominator model would hide important
database behavior.

Instead, I would use a controlled database change framework:

- One shared repository structure and release process.
- Engine-specific DDL directories, for example `postgres/`, `mysql/`, and
  `sqlserver/`.
- A common migration manifest with change id, owner, ticket, target engine,
  dependency, rollback strategy, and validation query.
- A standard migration runner per engine. This can be Liquibase where it fits,
  or native tools such as `psql`, `mysql`, and `sqlcmd` wrapped by
  Ansible/Jenkins.
- Common policy checks for naming, primary keys, audit columns, retention
  columns, destructive-change approval, and rollback notes.
- Engine-specific review for partitioning, indexes, stored procedures,
  functions, and views.

For simple application-owned schemas, Liquibase changelogs can help. For
non-trivial or high-throughput tables, I would keep explicit native DDL per
engine. Partitioning, indexes, and performance details are architecture
decisions, not only syntax.

Ansible should support the IaC tool in three areas:

- On-prem host bootstrap: packages, filesystem layout, kernel/sysctl settings,
  users, certificates, monitoring agents, backup agents, and audit agents.
- Database bootstrap: roles, least-privilege grants, standard schemas,
  extensions, SQL Server Agent jobs, maintenance jobs, and baseline checks.
- Operational automation: patch checks, backup restore tests, log shipping
  checks, and emergency runbooks.

To reduce drift, all modules or constructs should be versioned and promoted
through pull requests. CI should run the right checks for the selected tool:
CDK synth/diff, CloudFormation validation and Change Sets, or `terraform fmt`
and `terraform validate`. It should also run policy checks and Ansible
syntax/lint checks. Production changes should use reviewed plans or Change
Sets, immutable versions, and change tickets. Runtime drift should be checked
with CloudFormation drift detection, Terraform plan checks where Terraform is
used, and compliance scans.

For on-prem servers, Golden Images should be built by an image factory, not by
manual server setup. I would use Packer plus Ansible hardening to create base
images for Linux database hosts and Windows SQL Server hosts. Images should
include:

- OS baseline and CIS-aligned hardening.
- Required database engine prerequisites.
- Standard directory and filesystem layout.
- Monitoring, logging, EDR, audit, backup, and vulnerability scanning agents.
- Certificate trust stores and secure defaults.
- Local firewall baseline.
- No environment-specific secrets.

The Golden Image pipeline should be repeatable: build, scan, test, publish, and
promote. Each image should have a version, changelog, vulnerability scan result,
and smoke test. Environment-specific settings should be applied later through
the selected IaC tool and Ansible. This keeps the image clean and prevents
hidden one-off configuration. Existing on-prem servers should be compared with
the approved baseline. Non-compliant servers should be fixed with Ansible or
rebuilt from the current Golden Image.

### Data Purging Automation

The retention workflow should be scheduled centrally but executed close to each
database:

1. A scheduler starts the purge run. This can be Jenkins, EventBridge with
   Lambda, or Cron for isolated on-prem systems.
2. The job reads retention policy metadata by platform, schema, table, and date
   column.
3. It finds data older than 2 years.
4. For partitioned tables, it detaches or archives old partitions, validates
   counts or checksums, and drops the partitions.
5. For non-partitioned tables, it deletes rows in small ordered batches with
   throttling and lock timeout.
6. Each batch writes to an audit table with run id, actor, predicate, row count,
   start time, end time, and status.
7. Metrics and failures are sent to central monitoring.

The workflow should support dry runs, maximum rows per run, emergency stop, and
per-table exclusion windows during incidents or business events.

### JIT Production Access

The migration example is MySQL to PostgreSQL, but the company also has
production SQL Server in the core tier. JIT access should be a common security
process for all production databases, including SQL Server, PostgreSQL, and
MySQL.

For SQL Server, I would avoid permanent admin access for developers. The normal
setup should be:

- Developers have read-only or no direct access to production by default.
- A dedicated AD or Entra ID group exists for emergency SQL Server access, for
  example `prod-sqlserver-jit-fix`.
- This group is mapped to a SQL Server login.
- The login is mapped to a limited database role, not directly to `sysadmin`.
- `sysadmin` is allowed only for a break-glass process with stronger approval.
- SQL Server Audit is enabled for login events, role changes, permission
  changes, DDL, and DML on sensitive tables.

Example setup on SQL Server:

```sql
create login [DOMAIN\prod-sqlserver-jit-fix] from windows;

use [ProductionDB];
create user [DOMAIN\prod-sqlserver-jit-fix]
  for login [DOMAIN\prod-sqlserver-jit-fix];

create role [jit_bugfix_executor];
grant execute to [jit_bugfix_executor];
grant view definition to [jit_bugfix_executor];
alter role [jit_bugfix_executor]
  add member [DOMAIN\prod-sqlserver-jit-fix];
```

If the fix needs data changes, I would prefer a reviewed stored procedure or SQL
script instead of giving broad table access. For example, the DBA can create a
temporary stored procedure for the exact fix, grant `execute`, and remove it
after the incident.

The JIT workflow should be:

1. The developer opens a ticket with incident id, target SQL Server, database,
   requested action, reason, and required duration.
2. The service owner approves the business need.
3. The DBA or security approver checks the requested permission. If a script can
   solve the issue, the DBA asks for a reviewed script instead of admin access.
4. A PAM tool, Entra Privileged Identity Management, or an automation job adds
   the developer to the JIT AD/Entra group for a fixed time, for example 30 or
   60 minutes.
5. The session is audited. SQL Server Audit, login events, role changes, and
   executed statements are linked to the ticket id.
6. At expiry, automation removes the developer from the group. A separate check
   confirms that the user no longer has production access.
7. After the incident, the DBA reviews what was executed and attaches the audit
   evidence to the ticket.

The automation can be implemented as a Jenkins job, ServiceNow workflow, or
PAM/PIM workflow. The important part is that grant and revoke are automatic and
auditable. A simple SQL Server side table can also record the access window:

```sql
create table dbo.jit_access_audit (
  id bigint identity primary key,
  ticket_id varchar(100) not null,
  login_name sysname not null,
  database_name sysname not null,
  requested_role sysname not null,
  approved_by sysname not null,
  valid_from datetime2 not null,
  valid_until datetime2 not null,
  status varchar(30) not null
);
```

This process does not block the developer, because access can be granted quickly
after approval. At the same time, it meets compliance requirements because the
access is time-limited, approved, logged, and automatically revoked.
